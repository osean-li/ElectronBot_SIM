📖 为什么需要这个项目
ElectronBot 是稚晖君 2023 年开源的一款桌面双臂机器人——6 个舵机、全 3D 打印、巴掌大小。但要复现它，你得有打印机、买舵机、焊 ESP32、刷固件。更关键的是：强化学习训练需要数百万次交互，真机舵机几分钟就烧。碰撞、摔落、过度旋转——每个 AI 的试错在真机上都是物理损坏。ElectronBot-SIM 在 MuJoCo 里建了一个 1:1 的数字孪生：相同的 6 关节结构、相同的运动学约束、相同的控制接口——你今天在电脑前写的每一行代码，未来买到硬件时都直接可用。整条链路在一台 RTX 2060 上即可跑通。
核心概念：数字孪生 + MCP 分界线

ElectronBot-SIM 的设计围绕两条核心原则展开：

  • 数字孪生：MuJoCo 中的虚拟 ElectronBot 拥有与真机完全相同的 6 关节运动学链、舵机映射比、硬限位和安全裁剪逻辑。仿真行为 1:1 对齐真机固件源码（movements.cc），不做"优化"或"美化"。
  • MCP 是唯一分界线：AI 策略层只发送 JSON-RPC 命令，完全不感知下面是 MuJoCo 仿真还是 ESP32 真机。切换只需改一行代码：ElectronBotBackend("sim") ↔ ElectronBotBackend("cloud")。
数字孪生七层架构MCP分界线1:1对齐

1.1 问题：机器人开发的两道门槛

想在桌面机器人上跑 AI？你面前有两道门槛。

第一道：硬件门槛。ElectronBot 的 24 个 3D 打印零件、6 个舵机（SG90 + 2g + 4.3g）、一块 ESP32-S3 主控、焊接、固件刷写——对想入门机器人但手头没有硬件的人来说，任何一环都是死路。

第二道：安全门槛。即使你有了真机，强化学习需要数百万次试错。PPO 训练 100 万步，意味着 6 个舵机要来回转动数百万次——SG90 这种微型舵机的寿命通常只有几百小时，在 RL 训练中几分钟就可能过热烧毁。碰撞、摔落、关节极限——仿真里只是一行错误日志，真机上是物理损坏。

这两道门槛指向同一个答案：先在仿真里把策略训好，再部署到真机。 但这里又有一个新问题——仿真和真机之间通常存在巨大的"Sim2Real Gap"：仿真里的行为在真机上跑不起来，因为模型精度不够、物理参数不对、控制接口不兼容。

ElectronBot-SIM 的核心贡献，就是用 MuJoCo 1:1 复现 ElectronBot 的物理模型，用 MCP 协议统一仿真和真机的控制接口，把 Sim2Real Gap 压缩到最小。

1.2 七层架构：从 CAD 图纸到真机部署

项目代码按功能划分为四个独立包，对应一套 7 层逻辑架构：

L1 · 模型描述层

输入：ElectronBot.step（30.5MB，24 个零件）。经过 FreeCAD 按运动组合并导出→5 组 STL（base_link / body / head / left_arm / right_arm）→generate_inline_mesh.py 嵌入→单个 MJCF 文件（~1.7MB，零外部依赖）。这是全书的物理基础，第三章将完整走通这条链路。

L2 · 物理仿真引擎

MuJoCo 3.x 提供刚体动力学核心：7 个 body（运动学链从 base_link 到左右手指尖）、6 个 hinge 关节、6 个 position actuator（50Hz 控制频率，kp/kv 按舵机规格校准）。渲染后端同时支持 OpenGL（本地开发）和 EGL（headless 训练）。域随机化工具在此层注入物理参数噪声。

L3 · 动作系统层

ElectronBotActions 类封装了 12 种预设动作（举手/挥手/拍打/点头/转体等），以及舵机级精确控制接口 servo_move()。关键设计决策：线性插值而非缓动曲线——因为真机固件的 MoveServos()（movements.cc:87）用的就是线性插值，仿真必须 1:1 对齐。安全裁剪 ClampServoTarget() 复现了固件的 6 组硬限位。

L4 · MCP 协议层

这是架构中最关键的一层。McpSimBridge 实现了与真机完全一致的 MCP JSON-RPC 协议：method: "tools/call"，params.name: "self.electron.xxx"。注册了 12 个工具——其中 8 个与真机 release v2.2.6 完全对齐，4 个为仿真专属（servo_move / servo_sequences / home / get_ip）。McpSimBridge 同时兼容标准 MCP 嵌套格式和仿真内部扁平格式，方便调试。

L5 · 传感器与观测层

提供三种传感器——JointSensor（关节位置+速度）、CameraSensor（RGB+Depth+Segmentation，仿真专属）、ContactSensor（接触力）——由 ObservationBuilder 组装为两种观测模式：obs_mode="full"（研究用，含视觉和精确关节反馈）和 obs_mode="realistic"（Sim2Real 用，仅含真机可得数据：指令角度 + 运动状态 + 电池电压）。

L6 · 智能决策层

三条训练管线，难度递增：① BC 模仿学习——录制 12 种预设动作的专家轨迹，MSE 监督学习 50 epochs 得到初始策略；② PPO 强化学习——16-32 并行 MuJoCo 环境 + 域随机化，在 BC 策略基础上精调；③ 文本 VLA——LLM（如本地 Qwen2.5）将自然语言指令分解为 MCP 工具调用序列，行为树引擎（py_trees）执行。

L7 · 应用与评估层

Benchmark 套件包含 7 个标准任务（reach / push / pick_place / stack / follow / gesture × 5 项指标）。Sim2Real 部署提供三种模式：模式 A（云端 API 透传，立即可用）、模式 B（ONNX 本地推理，需固件 OTA）、模式 C（WebSocket 直连，<10ms 延迟，需固件 OTA）。真机校准工具 calibrate.py 将 6 维 trim 偏置写入 ESP32 NVS。

模块依赖关系

四个 Python 包的依赖方向严格单向：electronbot_benchmark → electronbot_ai → electronbot_sim → MuJoCo。其中 electronbot_sim（L2-L5）零 AI 依赖，可独立运行和测试。Sim2Real 包横跨 L4 和 L7，通过 Backend API 桥接仿真和云端。

1.3 本书阅读路线

本章为导论。后续六章按项目实际实施顺序展开，每一章覆盖一层或多层架构：

  • 第二章 · 环境搭建与首次模型加载（L1 入口）— 30 分钟跑通第一个仿真机器人
  • 第三章 · 从 STEP 图纸到 MJCF 模型（L1 核心）— 24 个零件变可仿真机器人
  • 第四章 · Gymnasium 环境与动作系统（L2+L3）— 封装标准 RL 环境 + 12 种动作
  • 第五章 · MCP 协议层的实现（L4）— AI 用同一句话驱动仿真与真机
  • 第六章 · AI 训练管线的实现（L5+L6）— BC → PPO → 域随机化
  • 第七章 · Sim2Real 部署实践（L7）— 策略零修改上真机 + 校准排错

每章末尾均附有"设计文档 vs 实际实现"差异表。这些差异不是 bug，而是工程实践中设计如何被现实修正的真实记录。

🔑 关键词释义
MuJoCoMulti-Joint dynamics with Contact，Google 维护的开源刚体动力学仿真器，使用 RK4 积分器，原生 Python 绑定。
MJCFMuJoCo XML 模型格式，定义 body（刚体）/ joint（关节）/ geom（几何体）/ actuator（执行器）的层级树。
MCPModel Context Protocol，基于 JSON-RPC 2.0 的 AI-工具交互协议。本项目用它桥接仿真与真机。
Sim2RealSimulation to Reality，从仿真训练的策略迁移到真实硬件。域随机化是缩小差距的主流方法。
GymnasiumOpenAI Gym 继任者，标准 RL 环境接口规范（reset / step / render / close）。
Inline Mesh将 STL 网格顶点数据直接嵌入 MJCF XML 中，实现单文件自包含（~1.7MB），零外部依赖。
📂 项目顶层代码组织
src/electronbot_sim/— 仿真核心包（L2-L5），零 AI 依赖，独立可测。包含 env.py / mcp_bridge.py / backend.py / actions/ / sensors/
src/electronbot_ai/— AI 训练包（L6），依赖 electronbot_sim。包含 il/（BC/ACT）/ rl/（PPO/SAC）/ vla/（LLM→MCP）/ tasks/（7 个标准任务）
src/electronbot_benchmark/— 评估包（L7）。suite.py / run.py / report.py，7 任务 × 5 指标标准化评估
src/electronbot_sim2real/— Sim2Real 部署包（L7）。deploy_cloud.py（云端）/ deploy_onnx.py（本地）/ deploy_websocket.py（直连）/ calibrate.py（校准）
assets/— 静态资产（L1）。cad/（STEP 图纸）/ meshes/（5 组 STL）/ mjcf/（MJCF 模型 + 场景文件）
⚙️ 实现逻辑
  1. 建模先行（L1）— ElectronBot.step → FreeCAD 按运动组合并 → 5 组 STL → generate_inline_mesh.py → MJCF 单文件
  2. 引擎驱动（L2）— MuJoCo 加载 MJCF → 7 body 运动学链 + 6 hinge 关节 + 6 position actuator → Gymnasium 封装
  3. 控制对齐（L3）— ElectronBotActions 1:1 复现固件 movements.cc → 线性插值 + 6 组硬限位 + 12 种预设动作
  4. 协议桥接（L4）— McpSimBridge 注册 12 个 MCP 工具 → 双格式兼容（标准 MCP + 扁平调试）→ Backend API 统一 sim/cloud
  5. 训练闭环（L5+L6）— BC 模仿学习初始策略 → PPO 强化学习域随机化精调 → VLA 文本规划 → Benchmark 评估
  6. 真机部署（L7）— 策略经 Backend API 零修改下放 → 云端透传（模式A）/ WebSocket 直连（模式C）/ ONNX 本地（模式B）→ 校准 tools
